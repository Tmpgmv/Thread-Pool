# Отчет по реализации CustomThreadPoolExecutor

## 1. Анализ производительности: сравнение с аналогами

Реализованная архитектура с множеством очередей имеет свои сильные и 
слабые стороны по сравнению с классическими реализациями из JDK или веб-серверов (Tomcat, Jetty).

### Сравнение со стандартным ThreadPoolExecutor
*   **Снижение Contention (конкуренции за блокировку):** 
* В стандартном `ThreadPoolExecutor` все потоки-продюсеры и потоки-воркеры конкурируют за `lock` одной единственной очереди. 
* Предоставленная реализация CustomThreadPoolExecutor существенно снижает эту конкуренцию, 
* так как продюсеры распределяют задачи по разным очередям. 
* Это дает выигрыш в пропускной способности (throughput) при огромном количестве одновременно поступающих
* мелких задач.
*   
* **Риск простоя воркеров (Голодание):** 
* Важной проблемой предоставленной реализации CustomThreadPoolExecutor является отсутствие механизма 
* *Work-Stealing* (кражи задач). Поскольку каждый воркер жестко привязан к своей очереди, 
* может возникнуть ситуация, когда одна задача выполняется очень долго, 
* блокируя свою очередь, в то время как другие воркеры быстро очистили свои очереди и простаивают. 
* Стандартный `ThreadPoolExecutor` лишен этого недостатка.

### Сравнение с ForkJoinPool
*   `ForkJoinPool` из стандартной библиотеки Java также использует множество очередей 
* (каждый поток имеет свою двустороннюю очередь - Deque).
*   Однако `ForkJoinPool` решает проблему простоя с помощью *Work-Stealing*: если поток 
* опустошил свою очередь, он идет и "крадет" задачи с конца очереди другого, занятого потока. 
* Предоставленная реализация производит балансировку *только при поступлении задачи*, 
* поэтому она будет проигрывать `ForkJoinPool` на неоднородных по времени выполнения задачах.

### Итог по производительности
Предоставленный `CustomThreadPoolExecutor` покажет **максимальную производительность** в сценариях, где:
1. Задачи поступают с очень высокой интенсивностью из множества разных потоков.
2. Время выполнения всех задач примерно одинаково (гомогенная нагрузка).

## 2. Мини-исследование: Оптимальные параметры пула

Производительность любого пула потоков критически зависит от четырех основных параметров. 
Максимальные показатели достигаются не установкой "наибольших" значений, 
а точным подбором под характер нагрузки (CPU-bound или I/O-bound).

**Формула Брайана Гетца для вычисления оптимального размера пула:**
$N_{threads} = N_{cpu} \times U_{cpu} \times (1 + \frac{W}{C})$
Где:
* $N_{cpu}$ — количество доступных ядер процессора.
* $U_{cpu}$ — целевая утилизация процессора (от 0 до 1).
* $\frac{W}{C}$ — отношение времени ожидания (Wait time, например I/O) к времени вычислений (Compute time).

### Сценарий 1: Вычислительные задачи (CPU-bound)
Для задач, которые активно используют процессор (математические расчеты, криптография, сортировка массивов):
*   `corePoolSize` и `maxPoolSize`: Должны быть равны количеству логических ядер процессора (или $N_{cpu} + 1$, чтобы избежать простоя при редких Page Faults). Избыток потоков приведет к деградации производительности из-за частого переключения контекста (Context Switch).
*   `queueSize`: Рекомендуется делать большой, чтобы аккумулировать всплески нагрузки.
*   `keepAliveTime`: Не имеет большого значения, так как потоки редко завершаются.

### Сценарий 2: Задачи ввода-вывода (I/O-bound)
Для задач, связанных с походами в БД, сетевыми запросами или чтением файлов (как в веб-серверах Tomcat/Jetty):
*   `maxPoolSize`: Должен быть значительно больше количества ядер (часто от 50 до 500). Потоки большую часть времени "спят", ожидая ответа от сети, поэтому процессору нужны другие активные потоки для работы в это время.
*   `corePoolSize`: Устанавливается на уровне 10-20% от максимального, чтобы экономить ресурсы ОС в периоды отсутствия трафика.
*   `queueSize`: Должен быть ограниченным (Bounded). Слишком большая очередь приведет к тому, что задачи устареют (пользователь закроет браузер), пока до них дойдет очередь. При переполнении лучше использовать механизм отказа (Reject).
*   `keepAliveTime`: Оптимально 30-60 секунд. Позволяет пулу быстро "сдуваться" после пиковых нагрузок, освобождая оперативную память ОС.

## 3. Принцип действия механизма распределения задач и балансировки

В отличие от стандартного `java.util.concurrent.ThreadPoolExecutor`, который использует единую глобальную блокирующую очередь (Blocking Queue) для всех задач, 
предоставленная реализация CustomThreadPoolExecutor использует **массив независимых очередей**. 
Каждому потоку (Worker) при создании назначается своя собственная очередь.

**Алгоритм балансировки (Least Loaded):**
При поступлении новой задачи через метод `execute()`, пул не отправляет ее в общую очередь. Вместо этого он перебирает все существующие очереди и оценивает их текущий размер. Задача помещается в ту очередь, 
в которой на данный момент находится меньше всего ожидающих задач.

Этот подход направлен на равномерное распределение нагрузки еще на этапе сабмита (подачи) задачи. 
Если пулу не хватает активных потоков для поддержания минимального резерва (`minSpareThreads`), он проактивно создает новые потоки вплоть до достижения `maxPoolSize`.
